CSS

Syntax

  • Benutze Soft-Tabs mit 2 Leerzeichen – es ist die einzige Möglichkeit zu garantieren, dass der Code in allen Umgebungen gleich gerendert wird.
  • Lasse beim gruppieren von Selektoren jedem Selektor seine eigene Zeile.
  • In Fällen, in denen ein Regelsatz nur eine einzige Deklaration enthält, kann dieser in einer Zeile mit dem Selektor stehen.
  • Für bessere Lesbarkeit, füge ein Leerzeichen vor jeder öffnenden Klammer eines Deklarationsblocks hinzu.
  • Platziere schließende Klammern in einer neuen Zeile.
  • Füge ein Leerzeichen hinter jedem : für jede Deklaration hinzu.
  • Für ein genaueres Fehlerreporting sollte jede Deklaration in einer eigenen Zeile stehen.
  • Beende jede Deklaration mit einem Semikolon. Für die letzte Deklaration ist dies optional.
  • Kommagetrennte Eigenschaftswerte sollten ein Leerzeichen nach jedem Komma enthalten (z.B.box-shadow).
  • Füge Eigenschaftswerten oder Farbwerten keine führenden Nullen hinzu (z.B., .5 anstatt 0.5 und -.5px anstatt -0.5px).
  • Schreibe Hexwerte klein, z.B. #fff. Kleingeschriebene Buchstaben sind beim Lesen eines Dokumentes einfacher zu unterscheiden, da sie mehr einzigartige Formen aufweisen.
  • Verwende kurze Hexwerte, wenn möglich. Z.B. #fff anstatt #ffffff.
  • Setze Attributwerte in Anführungszeichen, z.B. input[type='text']. Sie sind nur in einigen Fällen optional und es ist eine Gewöhnung an Konsistenz.
  • Vermeide das Setzen von Einheiten für Nullwerte, z.B. margin: 0; anstatt margin: 0px;.

Vermeide @import

Verglichen mit <link>s ist @import langsamer, führt zu mehr Seitenanforderungen und kann andere unvorhergesehene Probleme verursachen. Vermeide es und benutze stattdessen andere Ansätze:

  • Benutze mehrere <link> Elemente
  • Kompiliere das CSS mit einem Präprozessor wie Sass oder Less in eine einzige Datei
  • Verbinde die CSS Dateien mit Tools, die von Rails, Jekyll oder anderen Umgebungen bereitgestellt werden

Klassennamen

  • Schreibe Klassennamen in Kleinbuchstaben und benutze Bindestriche (keine Unterstriche oder camelCase).
  • Vermeide ausschweifende und beliebige Kurznotationen. .btn ist eine nützliche Abkürzung für button, aber .t ist bedeutungslos.
  • Benenne Klassen so kurz und bündig wie möglich.
  • Benutze aussagekräftige Namen; benutze zweck- oder strukturgebundene Namen, anstatt toll klingende.
  • Schreibe selbstsprechende Klassen mit dem Präfix .is-

Selektoren

  • Versuche Klassen statt generische Element-Tags zu nutzen, um eine optimale Renderingperformance zu erreichen.
  • Vermeide mehrere Attributselektoren (z.B. [class^="..."]) bei gewöhnlich verwendeten Komponenten. Es ist bekannt, dass die Browserperformance dadurch beeinträchtigt wird.
  • Halte Selektoren kurz und versuche die Anzahl der Elemente in jedem Selektor auf drei zu begrenzen.
  • Setze den Geltungsrahmen zum nächsten Elterntag nur wenn nötig (z.B. wenn keine Präfixklassen verwendet werden).

Aufbau

  • Ordne Codesektionen nach Komponenten an.
  • Entwickle eine konsistente Kommentierungshierarchie.
  • Benutze konsistente Leerzeichen zu deinem Vorteil, wenn einzelne Sektionen in einem größeren Dokument voneinander getrennt werden.
  • Wenn mehrere CSS Dateien verwendet werden, teile sie nach Komponenten, anstatt Seiten auf. Seiten können neu angeordnet und Komponenten verschoben werden.

Editor Einstellungen

Stelle deinen Editor folgendermaßen ein, um verbreitete Inkonsistenzen und dirty diffs zu vermeiden:

  • Benutze Soft-Tabs mit zwei Leerzeichen
  • Schneide nachlaufende Leerzeichen beim Speichern ab.
  • Aktiviere UTF-8 codierung.
  • Füge am Ende der Datei eine Leerzeile hinzu.

Ziehe in Erwägung, diese Einstellungen in die .editorconfig-Datei deines Projektes zu schreiben. Für ein Beispiel, schaue dir diese Datei in Bootstrap an. Mehr über EditorConfig erfahren.
